# 92期1隊 半自動勤排系統 — 全面品質改進設計

## 概述

使用迭代式 Agent Team 對勤排系統進行全面品質改進，涵蓋 6 個 KPI 維度，持續迭代直到所有維度達標。

## KPI 維度 & 通過標準

| # | KPI 維度 | 通過標準 |
|---|---------|---------|
| 1 | 功能正確性 | 全流程無 bug，邊界情況皆處理 |
| 2 | UX 易用性 | 操作流暢、提示清晰、響應式良好 |
| 3 | 程式碼品質 | 無重複邏輯、命名清晰、結構合理 |
| 4 | 安全性 | CDN SRI、輸入驗證、XSS 防護 |
| 5 | 效能 | 頁面載入 <2s、排班引擎 <1s |
| 6 | 可維護性 | 錯誤處理完善、資料持久化穩健 |

## Agent Team 架構

- **Team Lead**（主控）：協調任務、彙總結果、判定 KPI
- **QA-Tester**：Playwright 模擬使用者流程、E2E 測試
- **Code-Auditor**：程式碼審查、安全審計
- **Implementer**：修復問題、實作改進

## 改進項目（依優先級）

### 第一輪：高影響修復
1. CDN 加入 SRI integrity hash（安全性）
2. localStorage 資料驗證與 schema 檢查（可靠性）
3. 週數輸入 JS 防呆（功能正確性）
4. 全域錯誤處理 window.onerror（穩健性）
5. 備份/還原 JSON 功能（資料安全）

### 第二輪：中影響改進
6. ARIA 無障礙標籤（UX）
7. 鍵盤導航改進（UX）
8. 排勤空缺醒目提示強化（UX）
9. 操作確認與 undo 提示（UX）
10. 資料匯出確認流程（功能）

### 第三輪：品質打磨
11. _w 屬性汙染清理（程式碼品質）
12. 邊界情況強化（功能正確性）
13. 效能微調（效能）
14. 最終全面驗證（所有維度）

## 迭代流程

1. 分析 → 識別問題
2. 修復 → 實作改進
3. 驗證 → 重新測試
4. 評估 → 檢查 6 個 KPI
5. 通過 → 結束；未通過 → 重複
